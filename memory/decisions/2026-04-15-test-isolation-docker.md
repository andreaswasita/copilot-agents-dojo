# Test Isolation via Docker Compose

## Context

Integration tests were flaky due to shared database state between parallel test runs in CI.

## Decision

Each test suite spins up an isolated PostgreSQL container via Docker Compose with a unique port.

## Alternatives Considered

- **Shared DB with transaction rollback** — rejected: doesn't cover migration tests
- **SQLite in-memory** — rejected: behavior differences from production PostgreSQL

## Consequences

- Positive: Zero flaky tests from shared state, CI runs are fully reproducible
- Negative: ~30s overhead per suite for container startup
- Risks: Docker-in-Docker may not work in all CI environments
